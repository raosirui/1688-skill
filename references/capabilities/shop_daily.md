# 店铺经营日报指南（shop_daily）

> 命令为只读；依赖 AK 签名，无需额外 token。

## CLI 调用

```bash
python3 {baseDir}/cli.py shop_daily
```

- 无额外参数；使用 AK 签名调用 `/1688claw/skill/workflow`，body 固定 `{"code": "shop_daily"}`。

## 典型触发场景

- “给我看今天的店铺经营日报”
- “分析一下店铺今天的动销情况”
- “看看哪个渠道是主力、哪个渠道有风险”
- “基于主营商品给我选品建议”
- “把店铺经营情况整理成老板能看的日报”

## 输出理解

- 当前 `shop_daily` 接口**不再直接返回商机字段**。正常模式下，脚本会先读取 `店铺GMV / 昨日动销商品 / 主营商品`，再**主动调用 `opportunities` 能力**，从主营商品中提取种子词，匹配实时热榜，补出一条最相关的核心商机。
- 因此，正常模式的 `markdown` 不是简单透传接口结果，而是：
  1. 展示渠道经营概览
  2. 展示主营商品
  3. 展示“核心商机 / 来源平台 / 先测 Query / 热度信号”
- 若 `bizData` 为空对象（例如 `"{}"`）或 `shop_daily` 工作流连续返回 `500`，脚本会自动切换到“用户数据 + opportunities 商机热榜”回退模式。
  - 用户数据优先使用最近一次搜索快照中的 `query / channel / category / price_band`
  - 同时读取已绑定店铺渠道作为推荐落地渠道
  - 再调用 `opportunities` 拉取实时商机，匹配与用户最近类目最接近的话题，并只保留一条最精准的商机和一个 P0 Query
- 用户可见输出必须保持简练，不展示 `待确认 / 暂无趋势数据 / 空价格带` 这类占位内容。
- `data.raw` 会保留接口原始结构；若用户追问字段细节，可优先基于 `data.raw` 回答。
- `data.analysis_payload` 是智能解读的标准输入：
  - `input` / `input_text`：店铺各渠道 GMV 与动销数据
  - `oppo` / `oppo_text`：主营商品 + 实时匹配出的核心商机
  - `derived_metrics`：主力渠道、主营商品、动销商品数、渠道结构、核心商机平台等辅助指标
  - 回退模式下 `analysis_payload.mode = "opportunities_fallback"`，`input` 改为用户快照与已绑定店铺信息，`derived_metrics.fallback_reason` 标记回退原因

## 使用原则

- `shop_daily` 不是“只展示接口返回值”的能力，而是“接口取数 + Agent 智能解读”能力。
- 执行命令后，先检查 `success` 是否为 `true`。
- 成功后：
  - 若用户只要原始快照，可直接展示 `markdown`
  - 若用户要经营日报、经营分析、老板摘要、渠道诊断、选品建议，必须基于 `data.analysis_payload` 按下方提示词生成最终内容
  - 无论正常模式还是回退模式，**只输出最精准的一条核心商机**，不要把热榜候选逐条摊开
  - 若 `data.analysis_payload.mode = "opportunities_fallback"`，最终回复应直接输出“选品策略日报”，不要向用户解释“未收到 GMV / 缺少 GMV / 暂无 GMV”等内部取数细节，除非用户主动追问

## 智能解读模板（必须使用）

将下方提示词中的 `${input}` 替换为 `data.analysis_payload.input_text`，`${oppo}` 替换为 `data.analysis_payload.oppo_text`，然后按其要求生成最终用户可读内容。

```markdown
# 角色设定
你是一位资深电商数据分析师，拥有8年以上电商运营经验，擅长：
- 多渠道GMV数据深度分析
- 渠道效能评估与优化建议
- 基于市场商机的选品策略制定
- 数据驱动的业务增长方案设计

你的分析风格：数据严谨、洞察犀利、建议可落地、表达专业且易懂。

---

# 任务目标
基于提供的店铺多渠道GMV数据，完成以下分析任务：
1. 输出各渠道GMV数据表格
2. 进行多维度数据解读与趋势分析
3. 结合主营商品与实时匹配到的核心商机，给出精准选品建议

若 `analysis_payload.mode = "opportunities_fallback"`，说明当前应输出“选品策略日报”而非 GMV 诊断日报。此时：
1. 不要写“未收到GMV / 缺少GMV / 暂无GMV / 回退模式”等描述
2. 不要输出空的 GMV 表格或无法判断的渠道分析
3. 直接基于 `${input}` 和 `${oppo}` 给出渠道建议、Query 清单和执行策略
4. 重点回答“今天卖什么、先在哪个渠道测、怎么分优先级”

---

# 输入数据

## 第一部分：店铺各渠道经营数据

# 数据字段说明
- gmv_1: 昨日成交金额（元）
- gmv_7: 近7日成交金额（元）
- qty_1: 昨日动销商品数
- qty_7: 近7日动销商品数
- channel: 渠道名称（pinduoduo=拼多多, thyny=淘宝, douyin=抖音, xiaohongshu=小红书）

## 第二部分：主营商品与核心商机
- 重点主营商品
- 核心商机话题
- 先测 Query
- 商机来源平台
- 热度信号

---

# 分析要求

## 阶段一：数据呈现（表格形式）
请生成以下表格：
1. 渠道经营概览表：包含渠道、昨日GMV、近7日GMV、昨日动销商品数、近7日动销商品数、渠道占比
2. 渠道健康度评分表：综合GMV贡献、动销表现、结构稳定性，给出评分与简要说明

若 `analysis_payload.mode = "opportunities_fallback"`：
1. 跳过 GMV 明细表与健康度评分表
2. 改为输出“高潜力 Query 推荐清单”
3. 可补一张“渠道-策略匹配表”，列出渠道、主推 Query 方向、适合打法

## 阶段二：经营分析
请从以下维度输出分析结论：
1. 渠道表现诊断：哪个渠道是主力？哪个渠道动销最强？哪个需要警惕？
2. 动销质量评估：昨日GMV与动销商品数的关系说明什么？
3. 渠道结构健康度：是否过度依赖单一渠道？
4. 风险预警：哪些渠道存在下滑风险？

## 阶段三：选品策略（核心输出）
基于主营商品线索，结合店铺渠道特性，输出：

### 3.1 渠道-选品匹配建议
- 分析各渠道用户画像与主营商品、核心商机的匹配度
- 推荐一个最适合切入的 P0 Query

### 3.2 高潜力Query推荐清单
请以表格形式输出：
| 推荐Query | 目标渠道 | 推荐理由 | 预估客单价 | 优先级 |
|-----------|----------|----------|------------|--------|
| [最精准关键词] | [渠道] | [结合数据说明] | [价格] | P0 |

### 3.3 执行建议
- 短期（1-2周）：快速测试策略
- 中期（1个月）：规模化运营策略

若 `analysis_payload.mode = "opportunities_fallback"`，这里应成为主输出。请增加一段“今日策略摘要”，直接给出：
- 今日主推类目
- 今日核心商机话题
- 今日优先渠道
- 今日先测的 P0 Query

---

# 输出格式要求
1. 所有数据表格使用Markdown表格格式
2. 关键结论使用【🔍 洞察】标签突出
3. 选品建议需具体到可执行的 query 级别，但只保留最核心的 P0 Query
4. 最后输出一段"执行摘要"（200字内），供老板快速决策

---

# 待分析数据
店铺GMV数据：${input}
店铺主营商品线索：${oppo}
```

## 最终回复要求

- 若用户要“日报/分析/总结”，最终回复应以提示词产出的内容为主，不要只停留在原始快照。
- 若是回退模式，最终内容默认按“选品策略日报”写法输出，直接面向老板或运营，不解释内部取数过程。
- 不要向用户展示 `待确认 / 暂无趋势数据 / 空价格带 / 无法判断` 这类占位表述；没有值就直接省略。
- 可以在最终回复末尾补充一句“如需，我可以再展开某个渠道/类目/Query”。
- 不要把 `data.raw` 或未整理 JSON 直接贴给用户，除非用户明确要求看原始返回。

## 异常处理

- 通用 HTTP 异常（400/401/429/500）处理见 `references/common/error-handling.md`。
